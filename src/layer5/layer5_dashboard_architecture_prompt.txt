Layer 5 Dashboard Architecture Prompt
=====================================

**INSTITUTIONAL-GRADE TELEMETRY & PORTFOLIO DASHBOARD**

**Document Context**
Date: April 3, 2026
Scope: Complete redesign of Layer 5 dashboard following institutional standards
Target Audience: System architects, UX engineers, portfolio managers
Technologies: Dash/Plotly/React or native web stack (Python-optional)


EXECUTIVE SUMMARY
=================

Current State:
- Basic Dash app with limited KPI cards
- Simple pie and scatter plots
- No institutional risk metrics
- No regime or correlation visualization
- No trade forensics or attribution
- No real-time streaming perception
- Hardcoded layout unsuitable for large institutions

Target State:
- Multi-view institutional dashboard
- Real-time streaming updates with latency < 100ms perceived delay
- Comprehensive risk, performance, and regime dashboards
- Trade-by-trade forensics and attribution
- Model confidence calibration
- Drift detection and model health monitoring
- Comparable to TradingView, Bloomberg Terminal, Vanguard dashboard standards


STRATEGIC DESIGN PRINCIPLES
============================

1. **Trust Through Transparency**
   Users must understand exactly why trades were approved, vetoed, or skipped.
   Every decision point must be traceable with a clear audit trail.

2. **Real-Time Perception**
   Even if updates arrive in batches, the UI must feel live and responsive.
   Smooth transitions, instant feedback, no jarring delays.

3. **Expert Usability**
   Assume expert users (fund managers, risk officers, quants).
   Minimize cognitive load through smart defaults and expert shortcuts.
   Keyboard navigation, deep linking, export capabilities.

4. **Multi-Asset, Multi-Strategy Scalability**
   Support 10+ assets, 20+ strategies, 100+ concurrent live trades.
   Dashboard must not degrade or slow with scale.

5. **Loose Coupling with Layers 0-4**
   Display Layer 1, 2, 3, 4 outputs without reimplementing their logic.
   Mirror the data contracts, use the same Granularity keys.

6. **Compliance-Ready Audit Trail**
   Every user action, filter change, export, and export timestamp recorded.
   Support forensic analysis at 6-month or 3-year scale.


CURRENT STATE PROBLEMS & REQUIRED FIXES
========================================

**Problem 1: Missing Institutional Risk Metrics**
Current: Only approval_rate, avg_confidence, raw trade count
Required:
- Sharpe Ratio (daily, weekly, monthly)
- Sortino Ratio (downside deviation focus)
- Calmar Ratio (return relative to max drawdown)
- Maximum Drawdown (with date range identified)
- Underwater Plot (cumulative drawdown over time)
- Recovery Time (days to reach new high after drawdown)
- Win Rate, Profit Factor, Expectancy (in units of R)
- Consecutive Win/Loss Streaks
- Best and Worst Trade analysis
- Trade Duration statistics (avg hold, median, std dev)

**Problem 2: No Regime or Market Context Visualization**
Current: Trades shown in isolation, no regime overlay
Required:
- Regime label overlay on candlestick charts
- Regime transition markers
- Volatility (ATR) band on price chart
- ADX trend strength indicator
- Win rate by regime (e.g., "Trades in Trending_HighVol regime:" 65% win rate)
- Signal count by regime
- Regime-stratified performance metrics

**Problem 3: No Model Confidence Calibration**
Current: Single threshold (0.535) applied uniformly
Required:
- Calibration curve (predicted confidence vs actual win rate)
- Confidence distribution histogram
- False positive rate vs confidence threshold curve (ROC-style)
- Expectancy by confidence decile
- Precision-recall trade-off visualization
- Model drift alert (calibration shift over time)
- Per-granularity calibration (H1 vs H4 separate)

**Problem 4: No Trade Forensics or Attribution**
Current: Simple table of recent trades
Required:
- Detailed trade blotter with entry, exit, outcome
- Trade reason (which strategy, which regime, why ML vetoed if applicable)
- Entry and exit slippage
- Execution price vs theoretical price
- Commission and cost breakdown
- Attribution: how much P&L from each layer (0, 1, 2, 3, 4)
- Trade lifecycle: approved -> execution -> fill -> hold -> exit
- Heatmap of wins/losses by time-of-day

**Problem 5: No Correlation or Concentration Monitoring**
Current: No correlation data shown
Required:
- Rolling 30-day correlation matrix heatmap
- Current portfolio exposure (sum of position sizes by asset)
- Correlation to existing open positions for new signals
- Net delta exposure (EUR/GBP/JPY basket exposure)
- Concentration alerts (e.g., "3 correlated EUR trades pending")
- Correlation blocking events log

**Problem 6: No Strategy or Asset Attribution**
Current: Only approved/vetoed split shown
Required:
- Win rate and expectancy by strategy
- Win rate and expectancy by asset
- Win rate and expectancy by granularity (H1, H4, D1)
- Heatmap: strategy vs asset (e.g., Trend_EMA_ADX on EUR_USD)
- Strategy popularity (% of approved trades)
- Asset volatility and ATR comparison

**Problem 7: No Real-Time Streaming Perception**
Current: 5-minute refresh interval
Required:
- WebSocket or Server-Sent Events for sub-second updates
- Smooth transitions and animations
- Live price ticker (bid/ask) for each asset
- Trade execution animation (entry card -> execution card -> filled)
- Regime change flash/highlight
- New signal flash at the top before table insert

**Problem 8: No Model Health / Drift Monitoring**
Current: No observability into model behavior
Required:
- Model version and training date display
- Model threshold and feature contract info
- Training dataset size and date range
- Approval rate trend (daily, weekly)
- Feature importance ranking (top 10 for current model)
- Out-of-distribution signal detection (confidence near 0.5 cluster)
- Model performance on live data vs training data

**Problem 9: No Multi-View or Context Switching**
Current: Single monolithic dashboard view
Required:
- Tabbed interface: Overview, Risk, Regimes, Model, Trades, Assets, Strategies
- Customizable dashboard (drag-and-drop cards)
- Quick-link to individual strategy, asset, or trade forensics
- Drill-down from summary to detail

**Problem 10: No Institutional Export/Compliance**
Current: Only browser view
Required:
- Export all views to PDF (formatted)
- Export trade blotter to CSV/Excel
- Export performance report to PDF
- Audit log download (JSON)
- Screenshot annotations
- Watermark with timestamp and user name on exports

**TECHNICAL REQUIREMENTS**
==========================

**Frontend Framework:**
- Dash (Python) + Plotly for consistency with current stack
  OR
- React + D3 / Apache ECharts if new design requires performance optimization
- Mobile responsive (Tailwind CSS or Bootstrap 5)
- Dark mode toggle (institutional standard: dark mode default)

**Real-Time Data:**
- WebSocket connection to backend (Dash callbacks or custom server)
- Server-Sent Events (SSE) as fallback
- Message format: JSON with Timestamp, View, Deltas (partial updates)
- Max latency: 100ms perceived (allow 200ms network + processing)

**Backend API:**
- RESTful endpoints for data queries
- Streaming endpoint for real-time updates
- Caching strategy: Redis for frequently-accessed views
- Database queries: optimized to return views (pre-aggregated where possible)

**Data Source:**
- Primary: Fact_Live_Trades, Fact_Market_Regime_V2, Fact_Signals
- Secondary: Dim_Asset, Dim_Strategy, Dim_Strategy_Config, Fact_Market_Prices
- All queries must respect Granularity key (H1, H4, D1 filters)

**Performance Requirements:**
- Page load: < 3 seconds
- View switch: < 0.5 seconds (cached, animated)
- Real-time update render: < 100ms
- Support 50+ concurrent users without degradation
- Archive > 500k trades without slowdown

**Security:**
- HTTPS / TLS 1.3+
- Authentication: OAuth2 / SSO (LDAP / Azure AD)
- Authorization: role-based (viewer, analyst, trader, admin)
- Audit logging: all page views, exports, filter changes
- Data masking: redact sensitive PII if applicable
