Why Refactor Layer 4 To Consume Fact_Market_Regime End-To-End

You remove train-vs-live mismatch.
Layer 1 currently produces four labels, while Layer 4 is deriving a simpler binary regime internally in live_pipeline.py:162. That creates feature contract drift between what the model learned and what live trading feeds it.

You enforce one source of truth.
If regime context comes from one persisted table, every downstream system (training, inference, audit, dashboards) sees the same regime state for the same timestamp/asset/granularity.

You get proper multi-timeframe behavior.
Your system uses H1 and H4. Layer 4 should consume the already computed regime at the correct timeframe, not recompute a local approximation that may ignore granularity.

You improve reproducibility and auditability.
When a trade is approved/vetoed, you can trace exactly which stored regime label was used. This is essential for post-trade investigation and model governance.
You reduce hidden logic duplication.
Regime logic should live in Layer 1. Layer 4 should execute and risk-manage, not redefine market state. This keeps layers loosely coupled and easier to evolve.

You make scaling safer.
As assets increase beyond the current hardcoded set, central regime ingestion can scale dynamically from Dim_Asset while Layer 4 remains a clean consumer.
