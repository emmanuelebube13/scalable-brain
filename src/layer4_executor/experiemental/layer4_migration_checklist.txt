Layer 4 Migration Checklist
===========================

Scope
-----
This checklist tracks the migration of Layer 4 from a mixed signal/risk/execution script into a thin execution boundary aligned with Layers 0 through 3.

A. Architecture Boundary
- [ ] Remove any local regime derivation from live_pipeline.py.
- [ ] Remove any local signal generation from live_pipeline.py.
- [ ] Keep Layer 4 focused on execution, risk, correlation, and audit.
- [ ] Verify all market-state logic lives in Layer 1.
- [ ] Verify all signal-generation logic lives in Layer 2.
- [ ] Verify ML approval logic remains in Layer 3.

B. Model Artifact Contract
- [ ] Confirm the Layer 3 champion artifact naming convention.
- [ ] Replace the Layer 4 hardcoded model path with a stable deployment alias.
- [ ] Add support for a manifest or metadata file if needed.
- [ ] Preserve archived models in a versioned directory.
- [ ] Verify Layer 4 can load the current champion model reliably.
- [ ] Verify Layer 4 fails fast when the artifact is missing or stale.

C. Granularity Contract
- [ ] Ensure live records carry Timestamp, Asset_ID, Strategy_ID, and Granularity.
- [ ] Confirm Layer 4 uses the same granularity key as Layer 1 and Layer 3.
- [ ] Reject unsupported granularities explicitly.
- [ ] Do not mix D1 into the Layer 3 gatekeeper contract unless explicitly supported.
- [ ] Verify all logging and joins remain granularity-aware.

D. Risk Engine
- [ ] Keep ATR-based stop loss and take profit calculation.
- [ ] Remove any fixed pip stop logic if present.
- [ ] Validate stop loss and take profit are on the correct side of entry.
- [ ] Make risk settings strategy-aware if required by the promoted strategy.
- [ ] Skip trades when ATR or required price context is missing.

E. Correlation / Exposure Control
- [ ] Implement a rolling portfolio exposure check.
- [ ] Implement a rolling correlation gate.
- [ ] Block highly correlated new positions when exposure is already concentrated.
- [ ] Store correlation outcome in the trade log.
- [ ] Add a clear veto reason for correlation blocks.

F. Broker Execution
- [ ] Keep broker routing isolated from risk computation.
- [ ] Confirm OANDA execution uses rounded prices with correct precision.
- [ ] Confirm order submission is only attempted after all gates pass.
- [ ] Capture broker order id, execution time, and fill context.
- [ ] Handle broker failure without losing the trade context.

G. Logging and Audit
- [ ] Expand Fact_Live_Trades writes to include richer execution metadata.
- [ ] Record approved and vetoed trade decisions consistently.
- [ ] Log the exact reason for skipped or vetoed trades.
- [ ] Add slippage and correlation fields if the schema expects them.
- [ ] Ensure Layer 6 can reconcile execution records cleanly.

H. Upstream Compatibility
- [ ] Verify Layer 1 regime labels are read from Fact_Market_Regime_V2.
- [ ] Verify Layer 2 signals are read from the proper upstream source.
- [ ] Verify Layer 3 outputs are consumed as a stable artifact contract.
- [ ] Verify training and live feature contracts match on granularity.
- [ ] Verify no hidden assumptions depend on old binary regime logic.

I. Documentation
- [ ] Update doc/SYSTEM_ARCHITECTURE.md if Layer 4 responsibilities change.
- [ ] Update doc/ERD_ACTIVE_SCHEMA_2026.md if execution fields change.
- [ ] Update Layer 4 notes in src/layer4_executor/todo.txt after refactor.
- [ ] Add a short README or runbook for the new live pipeline if useful.

J. Validation
- [ ] Run a dry-run path with no broker execution.
- [ ] Confirm one approved trade flows through all checks.
- [ ] Confirm a regime lookup miss is handled cleanly.
- [ ] Confirm a granularity mismatch is rejected.
- [ ] Confirm a correlation veto is logged correctly.
- [ ] Confirm the stable model artifact loads correctly.
- [ ] Confirm archived model artifacts remain available for testing.

K. Rollout Order
- [ ] Implement the artifact contract first.
- [ ] Implement granularity preservation second.
- [ ] Remove duplicated regime and signal logic third.
- [ ] Add correlation gating fourth.
- [ ] Expand execution logging fifth.
- [ ] Update documentation last.

L. Completion Definition
- [ ] Layer 4 is a consumer of upstream truth, not a producer of market state.
- [ ] Layer 4 uses the current Layer 3 champion model through a stable path.
- [ ] Layer 4 supports archived model history for testing and audit.
- [ ] Layer 4 executes only after risk and correlation checks pass.
- [ ] Layer 4 writes audit-friendly logs with granularity intact.
- [ ] The live pipeline is easier to reason about than the current monolith.

Notes
-----
- Keep the change set minimal and focused.
- Prefer explicit contracts over implicit assumptions.
- Preserve backward compatibility where practical.
- Do not reintroduce legacy tables or old architecture patterns.
- If a schema change is required, update the docs in the same pass.
