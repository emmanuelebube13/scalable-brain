Layer 2 Strategy Map (Actual Runtime Execution)
===============================================

Last updated: 2026-04-05

Scope
-----
This document describes how Layer 2 actually executes strategy rules at runtime,
including buy/sell conditions and how conflicting rules are resolved.

Primary runtime source files:
- src/layer2_signals/generate_signals.py
- src/layer2_signals/signal_engine/core/engine.py
- src/layer2_signals/signal_engine/rules/evaluator.py
- src/layer2_signals/signal_engine/indicators/calculator.py
- src/layer2_signals/signal_engine/persistence/repository.py

Primary promotion/source mapping file:
- src/layer0/layer2_config_adapter.py


1) Runtime Truth Source
-----------------------
Layer 2 does NOT execute hardcoded strategy logic. It executes whatever active
rows exist in SQL strategy config tables at runtime:

- Dim_Strategy
- Dim_Strategy_Config
- Dim_Strategy_Asset_Mapping

For each selected granularity (default H1, H4), engine loads active configs where:
- s.Is_Active = 1
- c.Is_Active = 1
- m.Is_Active = 1
- c.Granularity = requested granularity
- Effective_To is null or future-dated

Important:
- "Actual execution" = current active Signal_Rules JSON in Dim_Strategy_Config.
- The strategy rules listed below are the shipped Layer 0 promotion defaults
	from layer2_config_adapter (what gets promoted when SQL seed scripts are applied).


2) Signal Semantics and Rule Engine Behavior
--------------------------------------------
Signal values:
- 1  = Buy
- -1 = Sell
- 0  = No signal

Condition operators supported:
- >, <, >=, <=, ==, !=
- cross_above: current(left) > right AND previous(left) <= right
- cross_below: current(left) < right AND previous(left) >= right

Previous-bar notation:
- "X.prev" resolves to previous bar value of X
- Example: STOCH_14.prev

Rule consolidation:
- Each rule returns -1, 0, or 1 per row.
- Engine sums all rule outputs per row, then applies sign:
	- sum > 0 -> final signal = 1
	- sum < 0 -> final signal = -1
	- sum = 0 -> final signal = 0
- If opposite rules trigger simultaneously, they cancel to 0.


3) Strategy Families and Executed Buy/Sell Rules

A) Trend_EMA_ADX
Variants: Trend_EMA_ADX_H1, Trend_EMA_ADX_H4, Trend_EMA_ADX_MultiTF

Trend_EMA_ADX_H1

- Sell: EMA_10 cross_below EMA_20 AND ADX_14 > 25

Trend_EMA_ADX_H4
- Buy: EMA_20 cross_above EMA_50 AND ADX_14 > 25
- Sell: EMA_20 cross_below EMA_50 AND ADX_14 > 25

Trend_EMA_ADX_MultiTF
- Buy: EMA_20 cross_above EMA_50 AND ADX_14 > 25
- Sell: EMA_20 cross_below EMA_50 AND ADX_14 > 25
- Note: Additional multi-timeframe confluence is documented as Layer 0 context,
	not fully encoded as extra Layer 2 rule conditions.


B) Trend_Donchian

Trend_Donchian_H1

4) Data Inputs, Warmup, and Persistence
---------------------------------------
Input prices:
- Fact_Market_Prices with columns Timestamp, Open, High, Low, Close, Volume

Warmup handling:
- Engine computes max indicator warmup period and drops warmup bars before
	evaluating rules.

Persistence:
- Non-zero signals are persisted to Fact_Signals using temp-table + MERGE.
- Upsert key includes Timestamp, Asset_ID, Granularity, Strategy_ID.


5) End-to-End Execution Flow
----------------------------
1. Load active strategy configs from SQL for requested granularity.
2. Fetch price bars from Fact_Market_Prices.
3. Calculate configured indicators.
4. Drop warmup rows.
5. Evaluate all configured rule conditions.
6. Consolidate rule outputs into final signal per row.
7. Persist non-zero signals into Fact_Signals.


6) Operational Notes
--------------------
- This file is a human-readable execution map; active SQL config remains source of truth.
- If promoted SQL configs differ from adapter defaults, runtime behavior follows SQL.
- Layer 2 is deterministic and rule-driven; ML filtering happens in Layer 3.
