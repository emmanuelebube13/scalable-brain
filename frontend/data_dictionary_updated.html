🎯 System Overview

Database Name: ForexBrainDB

Total Active Tables: 14 (7 Dimensions + 7 Facts + 2 Alternative Prices)

Deprecated Tables: 4 (scheduled for deletion)

Primary Key Strategy: Composite keys with Asset_ID as common hub

Data Integrity: Foreign keys enforced at DB level

Architecture Layers Supported

The schema supports the 6-layer execution pipeline:

Layer Purpose Key Tables
Layer 0 Strategy Qualification (Backtesting) Fact_Market_Prices, Fact_Trade_Outcomes
Layer 1 Market Regime Detection Fact_Market_Regime_V2
Layer 2 Signal Generation Dim_Strategy*, Fact_Signals, Fact_Indicator_Values
Layer 3 ML Meta-Labeling All three (3-way join)
Layer 4.5/5 Execution & Risk Fact_Live_Trades
Layer 6 Audit & Telemetry Fact_Live_Trades (outcomes)

📊 Dimension Tables (Reference/Lookup Data)

Dimension tables contain relatively static reference data that describe context for fact data.

Dim_Asset

Type: DIMENSION ACTIVE

Layer: Layer 0

Purpose: Master list of tradeable assets (currency pairs, commodities, etc.)

Read By: All layers (central hub)

Updated By: Manual via Python scripts (seed_dim_asset_test.py)

Column Definitions

Column Name Data Type Constraint Description
Asset_ID INT PRIMARY KEY Unique asset identifier (1, 2, 3, ...)
Symbol VARCHAR(20) NOT NULL, UNIQUE Currency pair symbol (EUR_USD, GBP_USD, etc.)
Market_Type VARCHAR(20) NULLABLE Asset class (FX, COMMODITIES, INDICES, BONDS)
Is_Active BIT NULLABLE 1 = actively traded, 0 or NULL = inactive
Created_Date DATETIME NULLABLE When asset was added to system

Sample Data

Asset_ID Symbol Market_Type Is_Active
1 EUR_USD FX 1
2 GBP_USD FX 1
3 USD_JPY FX 1

✅ Advantages

Central hub for all downstream fact tables; all price/signal/regime data references this
Simple surrogate key (Asset_ID) enables fast joins
Can support multiple asset classes (FX, commodities, indices)

⚠️ Disadvantages

No versioning if asset definitions change (e.g., if we split a pair)
Is_Active flag logic scattered across queries (should be enforced at DB level with CHECK constraint)
Market_Type is currently under-utilized; not used in portfolio correlation matrix calculations

🚀 Points of Improvement

  • Add CHECK constraint: Market_Type IN ('FX', 'COMMODITIES', 'INDICES', 'BONDS')
  • Add pip_value FLOAT and contract_multiplier FLOAT for true P&L calculations
  • Create indexed view v_Active_Assets that filters Is_Active = 1
  • Add deprecation_timestamp for soft-delete pattern
  • Track minimum historical bars required per asset before signal generation

Dim_Strategy

Type: DIMENSION ACTIVE NEW (April 2, 2026)

Layer: Layer 2

Purpose: Strategy identity hub (replaces deprecated Dim_Strategy_Registry)

Read By: Signal engine, ML gatekeeping, reporting

Parent Table: Dim_Strategy_Config, Dim_Strategy_Asset_Mapping reference this

Column Definitions

Column Name Data Type Constraint Description
Strategy_ID INT PRIMARY KEY Unique strategy identifier
Strategy_Key VARCHAR(50) NOT NULL, UNIQUE Human-readable name (e.g., 'bbands_mean_revert_v1')
Strategy_Type VARCHAR(30) NOT NULL Category (MEAN_REVERSION, TREND_FOLLOWING, MOMENTUM_OSCILLATOR)
Is_Active BIT NOT NULL 1 = enabled globally, 0 = disabled
Created_Date DATETIME NOT NULL When strategy was created
Updated_Date DATETIME NULLABLE Last modification timestamp
Python_Module VARCHAR(255) NULLABLE Import path (e.g., 'strategies.bbands_mean_revert')

✅ Advantages

Clean separation of strategy identity from configuration (enables versioning)
Supports multiple asset deployments of same strategy
Easy to enable/disable all instances of a strategy globally

⚠️ Disadvantages

No conflict detection if two strategies generate opposite signals on same asset/timeframe
Python_Module field is metadata-only (not actively validated)

🚀 Points of Improvement

  • Add signal_conflict_resolution (MAJORITY_VOTE, WEIGHTED_CONFIDENCE, HIGHEST_CONFIDENCE)
  • Add description NVARCHAR(MAX) for documentation
  • Add author VARCHAR(50) for accountability
  • Track version history with previous_strategy_id FK
  • Add approval workflow fields (approved_by, approved_date)

Dim_Strategy_Config

Type: DIMENSION ACTIVE NEW (April 2, 2026)

Layer: Layer 2

Purpose: Versioned parameter sets for each strategy. Contains JSON rules and indicator config

Child of: Dim_Strategy

Column Definitions

Column Name Data Type Constraint Description
Config_ID INT PRIMARY KEY Unique configuration identifier
Strategy_ID INT FOREIGN KEY References Dim_Strategy.Strategy_ID
Config_Version INT NOT NULL Incremental version (v1, v2, v3, ...)
Config_Timestamp DATETIME NOT NULL When this version was created
Config_Hash VARCHAR(255) NOT NULL SHA256 hash of JSON for deduplication
Indicator_Configs NVARCHAR(MAX) NOT NULL JSON: {"rsi_period": 14, "rsi_overbought": 70, ...}
Signal_Rules NVARCHAR(MAX) NOT NULL JSON: {"entry_logic": "rsi > 70", "exit_logic": "price < sma_50"}
Is_Active BIT NOT NULL 1 = currently used for signal generation
Backtest_Profit_Factor FLOAT NULLABLE Backtest PF (Gross Profit / Gross Loss)
Backtest_Expectancy_Per_Trade FLOAT NULLABLE Expected $ profit per trade (from backtesting)
Live_Trades_Count INT NULLABLE Number of live trades generated with this config
Live_Win_Rate FLOAT NULLABLE Real P&L win rate (0.0 to 1.0)

Sample JSON Content

Indicator_Configs: { "rsi_period": 14, "rsi_overbought": 70, "rsi_oversold": 30, "sma_fast": 9, "sma_slow": 50 } Signal_Rules: { "entry_long": "(rsi > 70) AND (close > sma_fast)", "entry_short": "(rsi < 30) AND (close < sma_fast)", "exit_mechanism": "atr_based_stop_loss", "time_to_live_hours": 12 }

✅ Advantages

JSON allows flexible, schema-less rule storage
Config_Hash enables reproducibility and deduplication
Tracks both backtest and real-time live performance metrics
Multiple versions per strategy without duplicating strategy table

⚠️ Disadvantages

JSON rules are unvalidated at DB layer (could have syntax errors)
Backtest metrics can drift significantly from live performance
No rollback mechanism if config introduces market drawdown
No audit trail of who approved/created each version

🚀 Points of Improvement

  • Add JSON_VALUE checks in INSERT/UPDATE triggers to validate JSON structure
  • Add approved_by VARCHAR(50) and approved_date DATETIME
  • Add max_consecutive_losses INT to auto-disable if exceeded
  • Create indexed view for active configs only
  • Add rollback_previous_config_id FK for quick version revert
  • Add sunset_date DATETIME for config expiration

Dim_Strategy_Asset_Mapping

Type: DIMENSION ACTIVE NEW (April 2, 2026)

Layer: Layer 2

Purpose: Maps strategy configs to specific assets and timeframes

Controls: Which strategies run on which assets/granularities

Column Definitions

Column Name Data Type Constraint Description
Mapping_ID INT PRIMARY KEY Unique mapping identifier
Strategy_ID INT FOREIGN KEY References Dim_Strategy.Strategy_ID
Asset_ID INT FOREIGN KEY References Dim_Asset.Asset_ID
Granularity VARCHAR(10) NOT NULL Timeframe (H1, H4, D1, W1)
Config_ID INT FOREIGN KEY References Dim_Strategy_Config.Config_ID (active version)
Priority INT NOT NULL Signal priority (1=highest, 99=lowest) for conflict resolution
Is_Active BIT NOT NULL 1 = enabled, 0 = disabled for this asset/granularity combo

Sample Data

✅ Advantages

Enables flexible deployment: same strategy on multiple assets/timeframes

Granularity control allows timeframe-specific trading rules

⚠️ Disadvantages

No correlation matrix to detect over-leveraged positions across assets
Priority is manual (should be AI-weighted based on live performance)
Can't limit max notional exposure per strategy/asset combo

🚀 Points of Improvement

  • Add estimated_correlation_to_existing_positions FLOAT (updated nightly)
  • Add ai_priority_weight_learned FLOAT (auto-adjusted based on live PnL)
  • Add max_notional_exposure FLOAT for position sizing limits
  • Add session_time_filter VARCHAR(100) (e.g., 'LONDON_08-12_UTC')
  • Create unique constraint: (Strategy_ID, Asset_ID, Granularity)
  • Track deployment_by VARCHAR(50) and deployment_date DATETIME

Dim_Indicator_Library

Type: DIMENSION ACTIVE

Layer: Layer 2

Purpose: Registry of all available technical indicators

Read By: Signal engine, indicator calculation pipeline

Column Definitions

Column Name Data Type Constraint Description
Indicator_ID INT PRIMARY KEY Unique indicator identifier

Category VARCHAR(50) NOT NULL Type (MOMENTUM, TREND, VOLATILITY, VOLUME, CUSTOM)
Required_Price_Fields VARCHAR(255) NOT NULL CSV of required fields: 'OHLCV', 'HLC', 'C', 'HL'

Sample Data

SMA TREND C Period 1
BOLLINGER_BANDS VOLATILITY C 20 1

✅ Advantages

Centralized indicator metadata enables reuse and documentation

⚠️ Disadvantages

No performance metrics (which indicators are most profitable)
Default parameters may be suboptimal for specific assets (e.g., RSI 14 vs RSI 21)
No tracking of which indicators are most frequently used

🚀 Points of Improvement

  • Add profitability_score FLOAT (win rate when used in signals)
  • Add optimal_parameter_set_per_asset NVARCHAR(MAX) (JSON from hyperparameter optimization)
  • Add update_frequency VARCHAR(50) (on_every_candle, on_london_open, on_market_open)
  • Add supported_granularities VARCHAR(255) (some indicators only valid on H4+)
  • Create view tracking indicator usage frequency
  • Add created_by VARCHAR(50) for custom indicators

📈 Fact Tables (Transactional Data)

Fact tables contain detailed, time-series data that capture business transactions and measurements.

Fact_Market_Prices

Type: FACT TABLE ACTIVE

Layer: Layer 0

Purpose: OHLCV candlestick data ingested continuously from OANDA

Data Retention: 18+ years of historical data

Ingestion Rate: Continuous (1H candles)

Updated By: ingest_oanda_prices.py

Column Definitions

Column Name Data Type Constraint Description
Price_ID BIGINT PRIMARY KEY Unique row identifier (identity)
Timestamp DATETIME NOT NULL Candle close time (UTC)
Asset_ID INT FOREIGN KEY References Dim_Asset.Asset_ID
Open FLOAT NOT NULL Opening price
High FLOAT NOT NULL Highest price in candle
Low FLOAT NOT NULL Lowest price in candle
Close FLOAT NOT NULL Closing price
Volume BIGINT NULLABLE Tick volume or transaction count
Granularity VARCHAR(10) NULLABLE Timeframe (H1, H4, D1, W1) - for ingestion traceability

Indexes

  • PRIMARY KEY: Price_ID
  • UNIQUE: (Asset_ID, Timestamp, Granularity) - for single-candle lookups
  • INDEX: (Timestamp DESC) - for resume mechanism in ingest
  • INDEX: (Asset_ID, Timestamp DESC) - for asset history traversal

✅ Advantages

18+ years of clean historical data supports backtesting and model training
Continuous ingestion ensures real-time data availability
Simple OHLCV schema is standard across all price data

⚠️ Disadvantages

Single Granularity column mixes H1/H4/D1 data (consider separate tables)
No bid/ask spread data (only mid-prices) reduces execution accuracy
Volume is tick volume, not contract volume (affects position sizing)
No liquidity metadata (bid-ask width, market depth)

🚀 Points of Improvement

  • Separate into Fact_Market_Prices_H1, H4, D1 tables (H4/D1 already done)
  • Add columns: Bid_Price, Ask_Price, Spread, Bid_Volume, Ask_Volume
  • Add Session VARCHAR(20) (TOKYO, LONDON, NY, SYDNEY) for session-based strategies
  • Add Data_Quality_Flag VARCHAR(20) (NORMAL, SPARSE, GAPPED) for gap detection
  • Create rolling hourly average of Bid_Ask spread for slippage simulation
  • Add open_interest INT if available from broker

Type: FACT TABLE ACTIVE NEW (April 2, 2026)

Layer: Layer 1

Purpose: Dynamically detected market regime (Trending_HighVol, Sideways_LowVol, etc.)

Read By: Layer 3 ML training (train_ml_gatekeeper.py)

Updated By: ingest_regimes.py and Fact_market_regime_v2.py

Column Definitions

Column Name Data Type Constraint Description

Asset_ID INT PRIMARY KEY, FOREIGN KEY References Dim_Asset.Asset_ID
Granularity VARCHAR(10) PRIMARY KEY Timeframe (H1, H4, D1)

  • Track regime change events in separate audit table

Fact_Signals

Type: FACT TABLE ACTIVE

Layer: Layer 2

Purpose: Raw trading signals (-1 SELL, 0 HOLD, 1 BUY) generated by strategy engine

Read By: Layer 3 ML training (train_ml_gatekeeper.py)

Updated By: Signal generation engine (Layer 2)

Column Definitions

Column Name Data Type Constraint Description
Timestamp DATETIME PRIMARY KEY Signal generation time (UTC)
Asset_ID INT PRIMARY KEY, FOREIGN KEY References Dim_Asset.Asset_ID
Granularity VARCHAR(10) PRIMARY KEY Timeframe (H1, H4, D1)
Strategy_ID INT PRIMARY KEY, FOREIGN KEY References Dim_Strategy.Strategy_ID
Signal_Value INT NOT NULL -1=SELL, 0=HOLD, 1=BUY
Strategy_Version VARCHAR(20) NOT NULL Config version used (v1, v2.1, etc.)
Config_Hash VARCHAR(255) NULLABLE SHA256 hash of exact JSON rules used (reproducibility)
Signal_Reason VARCHAR(500) NULLABLE Human-readable explanation (e.g., "RSI > 70 and Price > SMA50")
Rule_ID VARCHAR(100) NULLABLE Specific rule triggered (for audit trail)
Indicator_Snapshot NVARCHAR(MAX) NULLABLE JSON of all indicator values at signal time
Confidence_Score FLOAT NULLABLE Strategy's internal confidence (0.0-1.0), later overridden by ML

Sample Indicator_Snapshot JSON

{ "rsi_14": 75.3, "sma_9": 1.0875, "sma_50": 1.0823, "bollinger_upper": 1.0920, "bollinger_lower": 1.0780, "atr": 0.0045, "adx": 42.5, "volume_zscore": 1.8 }

✅ Advantages

Reproducible via Config_Hash (same signal always generated from same market state)
Rich audit trail with Signal_Reason and Indicator_Snapshot
Middle table in 3-way join for ML training (core to system)

⚠️ Disadvantages

Large JSON snapshots increase I/O during joins (main bottleneck for train_ml_gatekeeper.py)
Signal_Reason/Rule_ID are free-form text (impossible to aggregate or search)
No deduplication if same rule fires multiple times in same candle
No expiration time (stale signals should be invalidated after N bars)

🚀 Points of Improvement

  • Add signal_ttl_hours INT for automatic expiration (default 12)
  • Create Dim_Signal_Rules table for Rule_ID enumeration
  • Add ml_approval_score FLOAT post-facto (after Layer 3 runs)
  • Compress Indicator_Snapshot as VARBINARY instead of plain JSON
  • Create unique constraint preventing duplicate (Asset_ID, Timestamp, Granularity, Strategy_ID, Rule_ID)
  • Add is_filtered_by_ml BIT flag (whether ML gatekeeper was applied)

Fact_Trade_Outcomes

Type: FACT TABLE ACTIVE

Layer: Layer 0

Purpose: Backtesting labels (win/loss) for each signal. Used to train ML model

Read By: Layer 3 ML training (train_ml_gatekeeper.py - right table of 3-way join)

Updated By: evaluate_trades_atr.py (post-processing)

Column Definitions

Column Name Data Type Constraint Description
Timestamp DATETIME PRIMARY KEY Signal timestamp
Asset_ID INT PRIMARY KEY, FOREIGN KEY References Dim_Asset.Asset_ID
Strategy_ID INT PRIMARY KEY, FOREIGN KEY References Dim_Strategy.Strategy_ID
Signal_Value INT NOT NULL -1 (SELL), 1 (BUY) from original signal
Forward_Return FLOAT NULLABLE Pct return N bars after signal (e.g., 0.015 = +1.5%)
Is_Winner BIT NULLABLE 1 = PROFIT, 0 = LOSS (ground truth for ML)
Stop_Loss_Hit BIT NULLABLE 1 = yes, 0 = no (reached TP or expired)
Take_Profit_Hit BIT NULLABLE 1 = yes, 0 = no
Bars_Held INT NULLABLE How many bars before exit
Exit_Price FLOAT NULLABLE Price at which position was exited

✅ Advantages

Right table in 3-way join for ML training (critical for model)
Forward_Return is realized outcome (ground truth label for supervised learning)
Binary Is_Winner enables quick filtering and aggregation

⚠️ Disadvantages

Forward_Return is static (no mark-to-market updates during hold period)
Stop loss distance is hard-coded (doesn't adapt to regime changes)
Only binary win/loss (no "breakeven" category)
No transaction costs or slippage modeling

🚀 Points of Improvement

  • Add scenario_return FLOAT (mark-to-market at each candle during hold)
  • Add transaction_cost_pips FLOAT and slippage_pips FLOAT
  • Add atr_multiple_at_entry FLOAT (how many ATRs was SL away)
  • Create ternary outcome: LOSS, BREAKEVEN, WIN (instead of binary)
  • Add regime_at_entry VARCHAR(50) so ML can learn "BUY works best in Trending_HighVol"
  • Add max_drawdown_during_hold FLOAT for risk analysis

Type: FACT TABLE ACTIVE

Layer: Layer 2

Read By: Signal generation engine (Layer 2)

Column Definitions

Column Name Data Type Constraint Description

Asset_ID INT PRIMARY KEY, FOREIGN KEY References Dim_Asset.Asset_ID
Granularity VARCHAR(10) PRIMARY KEY Timeframe (H1, H4, D1)
Indicator_Instance VARCHAR(100) PRIMARY KEY Unique name (e.g., 'RSI_14', 'SMA_50', 'BBANDS_20_2')
Indicator_ID INT FOREIGN KEY References Dim_Indicator_Library.Indicator_ID

Indicator_Value_Lower_Band FLOAT NULLABLE For bands/envelopes (e.g., Bollinger lower band)
Config_Hash VARCHAR(255) NOT NULL Hash of indicator parameters (for reproducibility)
Calculation_Timestamp DATETIME NOT NULL When this value was computed

✅ Advantages

Caching improves signal generation speed (avoids recalculating every bar)

Separates indicator values from price data (requires JOIN to get full context)
One row per indicator creates very narrow, tall tables (50+ indicators per bar)
No versioning if calculation algorithm changes

🚀 Points of Improvement

  • Pivot/normalize: all indicators per bar in single row (reduces I/O)
  • Add previous_value FLOAT for momentum-of-indicator calculations
  • Add calculation_duration_ms INT to identify slow indicators
  • Create Dim_Indicator_Version table for algo change tracking
  • Add expiration flag for data older than lookback window
  • Materialized view for most recent indicator values only

Fact_Macro_Events

Type: FACT TABLE ACTIVE NEW

Layer: Layer 2

Purpose: Macroeconomic event surprises (NFP, CPI, etc.) extracted via NLP

Use Case: Context for signal filtering (avoid trading against major events)

Column Definitions

Column Name Data Type Constraint Description
id BIGINT PRIMARY KEY Unique event identifier
timestamp DATETIME NOT NULL Event release time (UTC)
asset_id INT FOREIGN KEY References Dim_Asset (e.g., USD-sensitive events)
event_title VARCHAR(255) NOT NULL Event name (e.g., 'US Non-Farm Payroll')
source VARCHAR(100) NULLABLE Data source (FRED, TRADINGECONOMICS, ECBDATAWAREHOUSE)
standardized_surprise_score FLOAT NULLABLE Z-score: (actual - expected) / std_dev (-3.0 to +3.0)
finbert_sentiment FLOAT NULLABLE News sentiment from FinBERT NLP (-1.0 to +1.0)
finbert_dispersion FLOAT NULLABLE Dispersion of sentiments across news sources (0.0-1.0)
cyclical_features NVARCHAR(MAX) NULLABLE JSON: {"economic_cycle": "expansion", "inflation_trend": "rising"}
inserted_timestamp DATETIME NOT NULL When this record was created (for audit)

✅ Advantages

Rich NLP-extracted features provide context for trading
Standardized surprise z-scores enable filtering
Sentiment analysis provides market direction bias

⚠️ Disadvantages

NLP sentiment is experimental (can be noisy, especially for non-English news)
Event-to-asset mapping is crude (same event affects many currency pairs differently)
No confidence scores from NLP models (FinBERT outputs probabilities but they're not stored)

🚀 Points of Improvement

  • Add event_category VARCHAR(50) (EMPLOYMENT, INFLATION, GDP, CB_RATE)
  • Add impacted_assets NVARCHAR(MAX) (JSON array which pairs affected)
  • Add market_reaction_5min FLOAT and market_reaction_1hour FLOAT
  • Add is_consensus_beat BIT (true if actual > expected for positive indicators)
  • Create embargo tracking (when event was locked/released under embargo)
  • Add FinBERT confidence scores
